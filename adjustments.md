# Narra — Recommended Project Structure

> **Stack:** Hono · TypeScript · Next.js 14 · Drizzle ORM · BullMQ · Qdrant · social-mcp  
> **Pattern:** Turborepo monorepo · pnpm workspaces · two apps + two shared packages  
> **Last updated:** April 2026

---

## Quick Reference

| Layer                  | App / Package     | Purpose                                                        |
| ---------------------- | ----------------- | -------------------------------------------------------------- |
| API + Workers + Agents | `apps/api`        | Hono server, MCP orchestration, BullMQ workers, social posting |
| Dashboard              | `apps/dashboard`  | Next.js 14, HITL UI, billing, analytics                        |
| Shared types           | `packages/types`  | TypeScript interfaces shared across both apps                  |
| Shared config          | `packages/config` | Zod env schemas, shared constants                              |

### 1. `apps/api/src/services/posting/` — Replace `ayrshare.ts` with `social-mcp.ts`

**Current:** `executor.ts` + `proxy-router.ts`  
**Required:** `executor.ts` + `proxy-router.ts` + `social-mcp.ts`

`social-mcp.ts` is the adapter between your workers and your own `social-mcp` npm package. It should be the **only** file in the codebase that imports from `social-mcp`. All other files go through this wrapper.

  // so executor.ts stays platform-agnostic

export async function fetchEngagementData(
  platform: Platform,
  accessToken: string,
  postId: string,
): Promise<EngagementData> {

`executor.ts` then calls `publishPost()` from this wrapper and never imports `social-mcp` directly. This keeps the dependency surface isolated to one file.

### 4. `apps/api/src/middleware/` — Add `plan-limits.ts`

**Current:** `auth.ts`, `validate.ts`  
**Required:** Add `plan-limits.ts`

This enforces plan quotas at the middleware layer before any route handler runs. Do not rely on `services/billing/limits.ts` alone — it will be missed on new routes.

```typescript
// middleware/plan-limits.ts

import { createMiddleware } from "hono/factory";
import { PlanLimitsEnforcer } from "../services/billing/limits";

export const requireLimit = (action: LimitAction) =>
  createMiddleware(async (c, next) => {
    const orgId = c.get("orgId");
    await PlanLimitsEnforcer.check(orgId, action); // throws 402 if over limit
    await next();
  });

// Usage in routes/brands.ts:
// app.post('/', auth, requireLimit('create_brand'), async (c) => { ... })
```

### 5. `apps/api/src/mcp/server.ts` — Use SSE transport, not stdio

**Current:** Unknown (confirm this)  
**Required:** Ensure you are using `SSEServerTransport`, not `StdioServerTransport`

`StdioServerTransport` takes over the process's stdin/stdout. When running inside a Hono HTTP server process, this breaks request/response handling. Use the SSE transport instead:

```typescript
// mcp/server.ts

import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { SSEServerTransport } from "@modelcontextprotocol/sdk/server/sse.js";

export function registerMcpRoutes(app: Hono) {
  const server = new McpServer({ name: "narra-agent-server", version: "1.0.0" });

  // Register all tools
  server.tool("retrieve_brand_context", retrieveBrandContextTool);
  // ... rest of tools

  app.get("/mcp/sse", async (c) => {
    const transport = new SSEServerTransport("/mcp/messages", c.res);
    await server.connect(transport);
  });

  app.post("/mcp/messages", async (c) => {
    // handle incoming MCP messages
  });
}
```

Import this in `services/billing/limits.ts` for credit deduction logic, and in `dashboard/components/billing/PlanCompareTable.tsx` for display. Keeping it in one place means a pricing change updates both automatically.

### 10. Remove root-level `config.ts`

**Current:** `config.ts` exists at monorepo root  
**Required:** Move its contents into `packages/config/src/index.ts` and delete the root file

Having config logic at the root level is confusing — contributors expect root to contain only tooling config (turbo, pnpm, tsconfig.base). Business logic config belongs in the package.

## Build & Dev Commands

# Install all workspaces
pnpm install

## Priority Order — What to Build First

Build in this order to unblock subsequent work:

1. `db/schema.ts` + `db/seed.ts` (plan_tiers) — everything else depends on the schema
2. `middleware/auth.ts` + `routes/auth.ts` — nothing else works without auth
3. `middleware/plan-limits.ts` — wire into all routes before they go live
4. `services/brand-ingestion/` — enables brand creation, which unblocks agents
5. `services/agent/` (orchestrator + all 3 agents) — core product value
6. `services/posting/social-mcp.ts` + `executor.ts` — connects agents to publishing
7. `workers/` (all 5 including overage) — enables async execution at scale
8. `services/billing/` (stripe + limits + usage-tracker + overage) — monetisation
9. `apps/dashboard/` — build components first, then compose into pages
10. `packages/config/credits.ts` + pricing display in dashboard
